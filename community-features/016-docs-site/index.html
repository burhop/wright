    Feature Brief: Documentation Site¶

    Create a dedicated documentation website for Wright using MkDocs Material, deployed to GitHub Pages. This replaces the scattered markdown files and PDF with a searchable, versioned, professional documentation site — the single most important signal of project maturity, as demonstrated by every popular agentic framework (OpenHands has docs.openhands.dev, CrewAI has docs.crewai.com, OpenClaw has docs.openclaw.ai).

    What to build¶

    Docs Framework Setup¶

    1. MkDocs Material project — Set up MkDocs with the Material theme in a docs/ directory (or a dedicated docs config):
    2. mkdocs.yml configuration at the repo root
    3. Material theme with Wright branding colors and logo
    4. Search plugin enabled
    5. Navigation organized by audience (users vs. contributors vs. API reference)

    6. Dark mode support (default to dark, matching engineering tooling aesthetics)

    7. GitHub Pages deployment (.github/workflows/docs-deploy.yml) — Auto-deploy docs on every merge to main:

    8. Build MkDocs site
    9. Deploy to GitHub Pages (gh-pages branch)
    10. The site should be available at https://burhop.github.io/wright/ (or a custom domain if configured)

    Documentation Structure¶

    Organize the docs site with these top-level sections:

    1. Getting Started — The first thing new users see:
    2. What is Wright? (elevator pitch + screenshot)
    3. Prerequisites (hardware, software)
    4. Quick Start with Docker (3-step copy-paste)
    5. Quick Start without Docker (local development)

    6. First steps after installation (what to try first)

    7. User Guide — For operators running Wright:

    8. Docker deployment guide (full version of the README's Docker section)
    9. Environment configuration reference (all env vars with descriptions and defaults)
    10. Makefile targets reference
    11. Backup and restore procedures
    12. Recovery runbook (migrate from docs/agent-docker-architecture.md)
    13. Health checking and monitoring

    14. Updating Wright (pulling new images, migration notes)

    15. Architecture — For developers understanding the system:

    16. System overview with Mermaid diagram
    17. Monorepo structure explanation (apps/, packages/)
    18. Agent adapter pattern
    19. Tool registry and MCP integration
    20. Local state and workspace file model (SQLite + mounted workspace files)
    21. Observability and tracing (OpenTelemetry + Jaeger)

    22. Migrate and consolidate content from: docs/technical_analysis.md, docs/agent-docker-architecture.md, constitution.md

    23. Contributing — For people who want to help:

    24. Development environment setup (detailed version)
    25. Spec-kit workflow guide (specify → plan → tasks → implement)
    26. Code style and conventions
    27. Testing guide (pytest, Playwright, smoke tests)
    28. PR submission guide

    29. Architecture decision records (link to specs/ directory)

    30. API Reference — Auto-generated from FastAPI:

    31. Embed or link to the FastAPI auto-generated OpenAPI docs (Swagger UI at /docs)
    32. Document how to access the API reference from a running instance

    33. If feasible, generate static API docs from the OpenAPI schema

    34. MCP Tools Catalog — Reference for available engineering tools:

    35. List of all MCP tool servers with descriptions
    36. How to add a custom MCP tool
    37. Tool online/offline fallback behavior
    38. Migrate relevant content from docs/Engineering MCP Tools Discovery.md

    Content Migration¶

    1. Migrate existing docs — Move content from scattered locations into the docs site:
    2. docs/technical_analysis.md → Architecture section
    3. docs/agent-docker-architecture.md → User Guide (Docker) + Architecture
    4. docs/Engineering MCP Tools Discovery.md → MCP Tools Catalog
    5. docs/virtual_engineer_architecture.pdf → Keep as downloadable but summarize key points in Architecture
    6. constitution.md → Referenced from Architecture and Contributing (keep original file, link to it)

    Polish¶

    1. Navigation and cross-linking — Ensure all pages are properly linked in the nav, with breadcrumbs and next/previous page navigation.

    2. Admonitions and callouts — Use Material's admonition blocks (note, warning, tip, danger) for important information throughout the docs.

    Constraints¶

    • Use MkDocs Material (Python-based, consistent with the project's tech stack) — not Docusaurus or other JS-based doc generators
    • Do not delete original source docs — the docs site should reference or migrate content, but original files can remain for backwards compatibility
    • The docs site must build and deploy without requiring any application code to be running
    • Keep the docs site lightweight — no heavy JavaScript, no complex build steps
    • The documentation must be maintainable by contributors who know Markdown (no custom templating languages)
    • Do not modify any application source code or Docker configuration
