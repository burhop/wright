    Feature Brief: Community Engagement Infrastructure¶

    Set up the communication channels, example projects, and marketing assets needed to attract users and contributors to Wright. This is the outward-facing "community building" work — the channels where people discover, learn about, and engage with the project beyond GitHub Issues and PRs.

    What to build¶

    Communication Channels¶

    1. Discord server — Create and configure a Wright community Discord server with channels:
    2. #announcements (read-only for members) — Release notes, breaking changes, project milestones
    3. #general — Open discussion about Wright, AI, mechanical engineering
    4. #support — Help with installation, configuration, and usage
    5. #contributing — Discussion for contributors (PRs, feature ideas, architecture questions)
    6. #showcase — Community members sharing what they've built with Wright
    7. #off-topic — Non-Wright conversation
    8. Create a server invite link and add it to: README.md, CONTRIBUTING.md, SUPPORT.md, and the docs site
    9. Set up basic moderation (rules channel, verification level, auto-mod for spam)

    10. Document the Discord setup and admin instructions so the project owner can manage it

    11. GitHub Discussions integration — Ensure GitHub Discussions (set up in feature 013) is cross-linked from:

    12. The Discord #support channel (pin a message directing complex questions to Discussions for better searchability)
    13. The README.md
    14. The docs site

    Example Projects¶

    1. Examples directory (examples/) — Create 3-5 self-contained example projects that demonstrate Wright's capabilities:
    2. examples/quickstart/ — Minimal "hello world" that connects to the API and runs a simple agent task
    3. examples/bracket-design/ — A mechanical bracket parametric design using the CAD tools
    4. examples/bolt-analysis/ — Bolt stress analysis using engineering calculation tools
    5. Each example should include:
      • A README.md explaining what it demonstrates and how to run it
      • Any necessary input files (CAD templates, material specifications)
      • Expected output or screenshots showing the result
    6. Examples should work with the Docker deployment (no local dev setup required)

    Launch Content¶

    1. Launch blog post — Write a markdown blog post (stored in docs/blog/ or as a draft for Dev.to/Medium):
    2. Title: "Introducing Wright: Your Local-First AI Mechanical Engineer"
    3. Sections:
      • The problem: Why mechanical engineers need local AI (IP protection, air-gapped networks, latency)
      • What Wright does: Multi-agent orchestration for CAD, FEA, and manufacturing
      • How it works: Architecture overview with diagrams
      • Getting started: Docker quickstart
      • What's next: Roadmap highlights
      • How to contribute: Link to GitHub and Discord
    4. Include screenshots or demo GIF

    5. Target length: 1500-2000 words

    6. Demo recording script — Write a script/outline for a 2-3 minute demo video:

    7. Scene 1: Show the Docker quickstart (15 seconds)
    8. Scene 2: Navigate the Wright web UI (15 seconds)
    9. Scene 3: Chat with the AI agent to design a bracket (60 seconds)
    10. Scene 4: Show the tool registry with MCP tools (15 seconds)
    11. Scene 5: Show the generated CAD output (15 seconds)
    12. Scene 6: Call to action — GitHub link, Discord invite
    13. Store the script in docs/demo-script.md

    Contributor Onramps¶

    1. "Good First Issue" curation — Create 5-10 GitHub Issues labeled good first issue that are:
    2. Self-contained (don't require understanding the full architecture)
    3. Well-described (include context, expected outcome, relevant files)
    4. Appropriately scoped (completable in 2-4 hours for a new contributor)
    5. Covering different skill sets: Python backend, TypeScript frontend, documentation, Docker, testing

    6. Store issue templates/drafts in docs/good-first-issues.md so the project owner can create them

    7. Contributor recognition — Set up contributor acknowledgment:

    8. Add an "all-contributors" configuration (.all-contributorsrc) or document how to use GitHub's built-in contributor graph
    9. Create a Contributors section in the README (can be auto-generated)
    10. Document the recognition policy in CONTRIBUTING.md

    Discoverability¶

    1. Awesome list submissions — Prepare submissions to relevant curated lists:
    2. Draft a submission PR description for awesome-ai-agents (or equivalent 2026 list)
    3. Draft a submission for engineering-focused awesome lists
    4. Draft a submission for Docker/self-hosted project lists
    5. Store drafts in docs/awesome-list-submissions.md with links to the target lists and their submission requirements
    6. Note: These should only be submitted AFTER the docs site and README overhaul are complete

    Project Metadata¶

    1. Star history badge — Add a star-history.com embed or badge to the README showing GitHub star growth over time. Store the embed code in the README branding feature (012) coordination doc.

    Constraints¶

    • Do not modify any application source code, Docker files, or CI/CD workflows
    • Discord setup must be documented as instructions (you can't create a Discord server via code)
    • Example projects should use the existing API — do not create new API endpoints for examples
    • The blog post is a draft — the project owner decides when and where to publish
    • Good First Issues are drafts — the project owner creates the actual GitHub Issues
    • Awesome list submissions should only happen after the repo meets their quality requirements
    • All content should be written in a professional, welcoming tone that encourages participation
