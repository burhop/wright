    Feature Brief: GitHub Templates & Issue Automation¶

    Set up GitHub Issue templates, Pull Request templates, and Discussion categories to standardize how the community reports bugs, requests features, and submits code contributions. This ensures maintainers receive structured, actionable information and contributors have clear guidance.

    What to build¶

    Issue Templates (YAML Forms)¶

    Create structured issue templates using GitHub's YAML form syntax (not markdown templates) for richer, more structured submissions:

    1. Bug Report template (.github/ISSUE_TEMPLATE/bug_report.yml) — A form-based template with fields for:
    2. Bug description (textarea, required)
    3. Steps to reproduce (textarea, required)
    4. Expected behavior (textarea, required)
    5. Actual behavior (textarea, required)
    6. Environment details (dropdown: Docker deployment / Local development / CI/CD)
    7. Wright version or commit SHA (input, required)
    8. Docker version if applicable (input, optional)
    9. Operating system (input, required)
    10. Relevant logs or screenshots (textarea, optional)

    11. Labels auto-applied: bug, needs-triage

    12. Feature Request template (.github/ISSUE_TEMPLATE/feature_request.yml) — A form-based template with fields for:

    13. Feature summary (input, required)
    14. Use case / problem description (textarea, required) — "What problem does this solve?"
    15. Proposed solution (textarea, required)
    16. Alternatives considered (textarea, optional)
    17. Which Wright component is affected (dropdown: API / Web UI / Agent Adapters / Tool Registry / Docker / MCP Tools / Other)
    18. Willingness to contribute (dropdown: Yes, I'd like to implement this / I'd like help implementing / I'm requesting this as a user)

    19. Labels auto-applied: enhancement, needs-triage

    20. Issue Chooser config (.github/ISSUE_TEMPLATE/config.yml) — Configure the issue type selector to:

    21. Show Bug Report and Feature Request as options
    22. Add a "Question / Help" option that redirects to GitHub Discussions Q&A category (not an issue template)
    23. Add a "Security Vulnerability" option that redirects to SECURITY.md (not a public issue)

    Pull Request Template¶

    1. PR template (.github/PULL_REQUEST_TEMPLATE.md) — A checklist that contributors see when opening a PR:
    2. Description of changes (what and why)
    3. Related issue number (e.g., "Closes #123")
    4. Checklist items:
      • [ ] I have read CONTRIBUTING.md
      • [ ] My changes follow the project's coding standards
      • [ ] I have added/updated tests for my changes (if applicable)
      • [ ] I have updated documentation (if applicable)
      • [ ] My changes do not introduce breaking changes
      • [ ] All existing tests pass (make test or equivalent)
      • [ ] My branch is up to date with the target branch
    5. For spec-kit features: additional checklist items:
      • [ ] Spec-kit artifacts (spec.md, plan.md, tasks.md) are updated
      • [ ] Changes comply with constitution.md

    GitHub Discussions¶

    1. Enable and configure GitHub Discussions with categories:
    2. Announcements (maintainers only) — Release notes, breaking changes, project updates
    3. Q&A (threaded, with answered marking) — Technical help and usage questions
    4. Ideas — Feature brainstorming before formal feature requests
    5. Show & Tell — Community members sharing what they've built with Wright
    6. Document the category setup instructions for the repo owner (Discussions are enabled via GitHub Settings, not files)

    Dependency Management¶

    1. Dependabot configuration (.github/dependabot.yml) — Automated dependency update PRs for:
    2. Python dependencies (pip ecosystem, check weekly)
    3. npm dependencies (check weekly)
    4. GitHub Actions (check weekly)
    5. Docker base images (check weekly)
    6. Group minor/patch updates into single PRs to reduce noise
    7. Set reasonable limits (max 5 open PRs per ecosystem)

    Labels¶

    1. Standard label set — Document a recommended set of GitHub labels for the repo owner to create:
    2. Priority: P0-critical, P1-high, P2-medium, P3-low
    3. Type: bug, enhancement, documentation, question, security
    4. Status: needs-triage, confirmed, in-progress, blocked, wontfix
    5. Component: api, web-ui, docker, agents, tools, ci-cd
    6. Contributor: good first issue, help wanted, hacktoberfest
    7. Include a script or markdown table documenting each label's name, color, and description

    Constraints¶

    • All templates must use GitHub's YAML form syntax (not markdown) for issue templates
    • Do not modify any source code, Docker files, or existing CI/CD workflows
    • Do not modify README.md or community health files (those are separate features)
    • The PR template should be lightweight enough that it doesn't discourage small contributions
    • Dependabot must not create excessive noise — use grouping and limits
