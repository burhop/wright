# Feature Specification: Engineering Workspace

**Feature Branch**: `005-engineering-workspace`

**Created**: 2026-06-02

**Status**: Draft

**Input**: User description: "We need to come up with the concept of an engineering workspace that is not unlike a git repo. All the users files for a particular set of tasks and agent context should be together in this workspace. If we run MCPs in our framework, final output files should end up in the workspace. Intermediate temp files can still go to temp. As with a Repo, we need to support viewing of files, abiltiy to create directories, move and delete files, and support git operations to support versions of the files. We shoudl have a default repo on the GB10, but should have an options page that allows us to specify github or other git servers. As much as possible, model off VS explorer."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Local Workspace Explorer (P1)

The user browses, edits, and organizes engineering files within an isolated task context. The sidebar panel is modeled after the VS Code Explorer, displaying a folder tree structure. Users can create folders, rename files, move files via drag-and-drop or context menus, and delete files. Clicking a file renders its preview (3D visualization for STL/CAD files, text/code editor for scripts/configs, and basic previews for JSON/data files). When the agent executes tools (e.g., OpenSCAD, simulation solver), all final output files are written to this workspace, while intermediate run files are directed to the system temp directory.

**Why this priority**: Core foundation of the engineering environment. It establishes the workspace layout and gives the user visual control over physical files and outputs.

**Independent Test**: Can be tested by creating a file and a folder via the UI explorer panel, checking that they appear on the local disk inside the workspace directory, and verifying that agent tools write final outputs there.

**Acceptance Scenarios**:
1. **Given** an active chat session, **When** the user clicks the "New Folder" icon in the explorer, **Then** a new folder is created in the directory and displays in the tree.
2. **Given** files generated by a simulation tool, **When** the tool finishes running, **Then** final mesh and result files appear in the workspace tree, and temporary log/solver files are absent.
3. **Given** a file in the tree, **When** the user drags it into a subdirectory, **Then** the file is moved on disk and the UI tree refreshes immediately.

---

### User Story 2 - Workspace Version Control & Git History (P2)

The user tracks file iterations and manages changes over time using local Git version control. The UI displays file status markers (e.g., Untracked `U`, Modified `M`, Clean) next to items in the explorer tree. A "Version Control" tab displays staged/unstaged changes, allows entering a commit message, committing changes locally, and viewing a linear history of past commits. Users can click a modified file to view a visual diff, or choose to revert a file to its state in the last commit.

**Why this priority**: Crucial for tracking iterations. Mechanical engineering involves parametric updates where rollback and version comparison are essential.

**Independent Test**: Can be tested by modifying a file, checking that it shows as `M` in the explorer tree, reviewing the diff, committing it, and verifying the commit appears in the history timeline.

**Acceptance Scenarios**:
1. **Given** a modified file in the workspace, **When** the user views the Version Control panel, **Then** it shows the unstaged changes and allows viewing a side-by-side or inline diff.
2. **Given** unstaged changes, **When** the user types a commit message and clicks "Commit", **Then** a local Git commit is created, the tree status returns to clean, and the commit history list updates.
3. **Given** a modified file, **When** the user clicks "Revert", **Then** the file is restored to its HEAD state and the status badge disappears.

---

### User Story 3 - Remote Git Syncing & Options Configuration (P2)

The user wants to sync their workspace with a remote repository hosting service (like GitHub or a private GitLab instance). They open the "Workspace Options" page to configure a remote URL and authentication credentials (e.g. personal access tokens). If configured, the Version Control panel displays "Push" and "Pull" actions. By default, if no remote is configured, the workspace initializes a local-only Git repository on the GB10 machine (offline-first).

**Why this priority**: Allows team collaboration and cloud backup, while maintaining full offline-first functionality by default.

**Independent Test**: Can be tested by entering a GitHub repository URL and personal access token in the settings, creating a commit, clicking "Push", and verifying the commit is pushed to the remote server.

**Acceptance Scenarios**:
1. **Given** a newly created workspace session, **When** no remote settings are specified, **Then** a local Git repository is initialized on the GB10 host and operates offline.
2. **Given** valid remote configurations in Settings, **When** the user clicks "Push", **Then** the local commits are pushed to the remote git branch.
3. **Given** an unreachable remote repository (due to network disconnection), **When** the user attempts to sync, **Then** the system fails gracefully with a clear offline warning, allowing the user to continue working locally.

---

### Edge Cases

- **Large Engineering Files**: Large datasets (such as 100MB+ FEA results or binary CAD models) can crash Git if tracked directly. The system must support auto-configuring a `.gitignore` or using Git LFS when these files are detected.
- **Concurrent File Lock**: If an agent is actively writing a 3D model file, and the user tries to rename or delete it, the system must show a busy/locked message and reject the modification until the process finishes.
- **Merge Conflicts**: If changes are pulled from a remote repository that conflict with local edits, the UI must display a conflict resolution panel allowing the user to choose "Keep Local", "Keep Remote", or "Merge".

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST support an isolated file system directory representing the active workspace for each session.
- **FR-002**: The UI MUST render a folder explorer tree showing files and folders matching the local directory.
- **FR-003**: The UI MUST allow creating directories, deleting files/folders, and renaming/moving files.
- **FR-004**: Agent MCP executions MUST direct final outputs to the workspace directory and intermediate files to `/tmp`.
- **FR-005**: The system MUST initialize a local Git repository inside the workspace directory upon creation.
- **FR-006**: The UI MUST show Git status indicators (`M` for modified, `U` for untracked) in the explorer tree.
- **FR-007**: The system MUST support local Git commits, file reverts, diff views, and commit history lists.
- **FR-008**: The system MUST provide an Options page to configure git remote credentials and remote sync (push/pull).
- **FR-009**: The system MUST operate locally on the GB10 by default, bypassing remote calls if no remote is configured.

### Key Entities

- **EngineeringWorkspace**:
  - `workspace_id` (UUID): Unique ID.
  - `session_id` (UUID): Linked chat session.
  - `local_path` (String): Directory path on disk.
  - `git_remote_url` (String | Null): Optional remote endpoint.
  - `created_at` (Timestamp)
- **GitCommit**:
  - `commit_hash` (String)
  - `message` (String)
  - `author` (String)
  - `timestamp` (Timestamp)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The workspace folder tree structure must render in under 150ms on first load.
- **SC-002**: File operations (create, rename, move, delete) must be performed and updated in the UI in under 100ms.
- **SC-003**: The local Git commit action must complete and clear status markers in under 500ms.
- **SC-004**: When offline, remote sync failures must be detected and report a graceful warning banner in under 1.5 seconds without hanging the UI.

## Assumptions

- **A-001**: The local GB10 machine has a standard git binary installed and accessible in the system path.
- **A-002**: Workspaces are bounded to a single active branch (typically `main`) for simplification in v1.
- **A-003**: Users have write permissions in the directory specified for their workspace store.
