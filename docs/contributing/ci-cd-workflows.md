# GitHub Workflows and CI/CD Pipeline

This document outlines the automated CI/CD workflows that execute when code is pushed or pull requests are created against the `dev` and `main` branches of the Wright repository.

---

## 1. Pipeline Overview

We configure six primary GitHub Actions workflows under `.github/workflows/` to handle code quality, container builds, smoke testing, and deployments:

| Workflow File | Purpose | Triggers | Core Steps / Outcomes |
| :--- | :--- | :--- | :--- |
| **`python-quality.yml`** | Python Quality Gate | Push or PR to `main` and `dev` | Runs Ruff linting and formatting checks, MyPy type checks, and Python `pytest` suite. |
| **`frontend-quality.yml`** | Web Client Quality Gate | Push or PR to `main` and `dev` | Runs ESLint, Prettier formatting checks, and TypeScript (`tsc`) type checking. |
| **`docker-build.yml`** | Container Build & Smoke Test | Push to `main`/`dev`/`v*` tags (on code changes), or any PR to `main`/`dev` | Builds the local container image, performs a non-blocking Trivy vulnerability check, runs a robust smoke test container check, and pushes the image on push events. |
| **`release.yml`** | Production Release Pipeline | Push to tag matching `v*` | Builds the production container, pushes tags (`v*` and `latest`) to Docker Hub, syncs `docker/DOCKER_HUB_README.md` to Docker Hub, and publishes the GitHub Release. |
| **`docs-deploy.yml`** | Documentation Publishing | Push to `main` or manual trigger | Builds the MkDocs material site and deploys it to GitHub Pages. |
| **`release-drafter.yml`** | Release Changelog Drafter | Push to `main` and `dev` | Automatically generates and updates a draft release compiling release notes from merged PRs. |

---

## 2. Push Events on `dev` and `main`

### Pushes to the `dev` Branch
1.  **Code Validation**: `python-quality` and `frontend-quality` run in parallel to verify syntax, formatting, and tests.
2.  **Container Build**: `docker-build` builds the container image `wright-agent:${{ github.sha }}` and runs the automated smoke test. If successful, it pushes the image to Docker Hub with the `dev` tag.
3.  **Release Notes**: `release-drafter` updates the draft release changelog on GitHub.

### Pushes to the `main` Branch
1.  **Code Validation**: `python-quality` and `frontend-quality` execute in parallel.
2.  **Container Build**: `docker-build` builds the image, runs the smoke test, and pushes the image to Docker Hub with both the `latest` and commit SHA tags.
3.  **Documentation Site**: `docs-deploy` runs to rebuild and publish the live documentation site hosted on GitHub Pages.
4.  **Release Notes**: `release-drafter` compiles the draft release notes.

## 3. Docker Hub Registry Authorization

To publish container images to Docker Hub during branch push workflows, the repository must be authorized.

### Required Repository Secrets
In your GitHub repository settings under **Settings -> Secrets and variables -> Actions**, configure the following secrets:
*   `DOCKERHUB_USERNAME`: Your Docker Hub account username.
*   `DOCKERHUB_TOKEN`: A personal access token (PAT) generated from your Docker Hub settings with read and write scopes.

*Note: For Pull Requests, the login and push steps are automatically skipped. The container is built and smoke-tested solely in the runner environment to verify PR correctness without publishing.*

## 4. Verification and Smoke Testing

Our CI pipeline includes an automated **Smoke Test** in `docker-build.yml` that validates container execution before pushing.

The test script:
1.  Spins up the newly built container with placeholder credentials.
2.  Polls the FastAPI health endpoint `/api/health` until responsive.
3.  Queries the agent health proxy `/api/agent/health` to verify the Hermes agent is connected.
4.  Triggers a workspace creation request (`/api/workspace/create`) to confirm SQLite database connections and git workspace structures initialize correctly.
5.  Queries `supervisorctl status` inside the container to ensure the `wright-api` and `hermes-webui` server processes are running.

### Local Quality Gate Check
Before committing and pushing changes, you can verify your code style and unit tests locally using the Makefile:

```bash
# Run Ruff linting, formatting, MyPy type checks, and pytest
make check

# Auto-format all code
make format
```

## 5. Next Steps for Maintainers

1.  **Activate GitHub Pages**: Enable GitHub Pages under your repository settings, and set the build source to deploy from the `gh-pages` branch.
2.  **Input Docker Hub Secrets**: Configure the `DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN` secrets to enable image pushing.
3.  **Review Release Drafts**: When ready to release, publish the draft release generated by `release-drafter` on the GitHub Releases page.
